# Database Design

> **Project:** Tamper-Evident Append-Only Audit Log System  
> **Database:** PostgreSQL  
> **ORM:** Prisma ORM

---

# 1. Overview

This document describes the database design for the Tamper-Evident Append-Only Audit Log System.

The primary goal of the database is **not simply to store events**, but to preserve their integrity and ensure they remain verifiable over time.

Unlike traditional audit tables where historical records can potentially be modified or removed, this database is intentionally designed around an **append-only architecture**.

Every record is permanently linked to the previous record using cryptographic hashing, creating an immutable chain that allows the application to detect any unauthorized modification.

---

# 2. Design Goals

The database is designed to satisfy the following engineering goals:

- Store immutable audit records.
- Prevent modification of historical logs.
- Prevent deletion of historical logs.
- Detect tampering through cryptographic verification.
- Support efficient querying by actor and time range.
- Remain simple, maintainable, and production-oriented.

---

# 3. Database Strategy

The current implementation intentionally uses **a single table**.

For the scope of this assignment, introducing additional tables would add unnecessary complexity without improving the core functionality being evaluated.

Authentication is handled through an API key stored in environment variables rather than a dedicated database table.

This keeps the project focused on the primary objective:

> Building a trustworthy append-only audit logging system.

---

# 4. Entity Overview

The database contains one primary entity.

## LogEntry

Each record represents one immutable audit event.

Once inserted, the record must never be modified or deleted.

Every new entry references the cryptographic hash of the previous entry, forming a continuous verification chain.

---

# 5. Table Structure

| Field | Type | Required | Description |
|--------|------|----------|-------------|
| id | UUID | ✅ | Unique identifier for each log entry |
| actor | String | ✅ | User or system responsible for the action |
| action | String | ✅ | Action performed |
| payload | JSONB | ✅ | Additional event information |
| previousHash | String | ✅ | Hash of the previous log entry |
| hash | String | ✅ | SHA-256 hash for the current entry |
| createdAt | Timestamp | ✅ | Time the event was created |

---

# 6. Field Design Decisions

## id

A UUID is used instead of an auto-incrementing integer.

Reasons:

- Better suited for APIs.
- Does not expose the total number of records.
- Independent of database sequence generation.
- Easier to scale in distributed environments.

---

## actor

Represents the user or system responsible for generating the event.

Examples:

- admin
- doctor
- receptionist
- system

This field is indexed because export operations frequently filter by actor.

---

## action

Represents the operation that occurred.

Examples:

- LOGIN
- CREATE_PATIENT
- UPDATE_PROFILE
- VERIFY_CHAIN

The application treats actions as descriptive event names rather than database operations.

---

## payload

Payload stores additional event information.

The structure varies depending on the event type.

Example:

```json
{
  "patientId": 101,
  "department": "Cardiology"
}
```

PostgreSQL **JSONB** is used because it:

- Supports structured JSON.
- Allows future querying if needed.
- Remains flexible without changing the schema.

---

## previousHash

Stores the SHA-256 hash of the previous log entry.

This creates the connection between two consecutive records.

The first record uses the predefined value:

```
GENESIS
```

---

## hash

Stores the SHA-256 hash calculated from the current record.

The value is unique for every record.

Any modification to the stored data changes the generated hash and immediately breaks the verification chain.

---

## createdAt

Stores the creation timestamp.

The timestamp is automatically generated by the database.

It is used for:

- chronological ordering
- export filtering
- auditing
- verification

---

# 7. Hash Chain Design

Every log entry references the previous log.

```
Entry 1

Previous Hash

GENESIS

↓

Current Hash

A1B2C3...

↓

Entry 2

Previous Hash

A1B2C3...

↓

Current Hash

F8D7E6...

↓

Entry 3

Previous Hash

F8D7E6...

↓

Current Hash

Z9Y8X7...
```

Changing any stored record causes every following hash to become invalid.

This property makes the audit trail tamper-evident.

---

# 8. Hash Generation

The application generates hashes using SHA-256.

Each hash is calculated from:

- Previous Hash
- Actor
- Action
- Serialized Payload
- Created Timestamp

Using a deterministic order ensures that the same input always produces the same output.

---

# 9. Database Constraints

The following constraints protect data integrity.

| Constraint | Purpose |
|------------|---------|
| Primary Key | Every log has a unique identifier |
| NOT NULL | Required fields cannot be empty |
| UNIQUE(hash) | Prevent duplicate hashes |
| Append-only | No UPDATE or DELETE operations |

---

# 10. Indexing Strategy

The following indexes improve query performance.

| Field | Reason |
|--------|--------|
| id | Primary key lookups |
| actor | Export filtering |
| createdAt | Date range filtering |
| hash | Verification and uniqueness |

# 11. Engineering Decisions

### Why PostgreSQL?

- ACID compliant.
- Reliable transactions.
- Native JSONB support.
- Excellent production support.
- Preferred by the assignment.

---

### Why Prisma?

- Type-safe queries.
- Easy migrations.
- Strong TypeScript integration.
- Improved developer experience.

---

### Why JSONB?

Payload structures vary between events.

Using JSONB avoids unnecessary schema changes while preserving structured data.

---

### Why UUID?

UUIDs avoid exposing record counts and provide globally unique identifiers suitable for production APIs.

---

### Why Only One Table?

The assignment evaluates the integrity of the append-only log.

Introducing additional tables for users or API keys would increase complexity without improving the primary objective.

Authentication is therefore handled using an API key stored in environment variables.

---

# 12. Assumptions

The following assumptions guide the current implementation.

- Log entries are immutable.
- Every request creates a new log entry.
- Historical records are never modified.
- Historical records are never deleted.
- Verification is performed sequentially.
- The application uses a single PostgreSQL instance.
- API key authentication is sufficient for this assignment.

---

# 13. Future Improvements

Possible production enhancements include:

- Multiple API keys stored in the database.
- User and role management.
- Merkle tree verification.
- Database partitioning.
- Read replicas.
- Cryptographic signatures.
- Soft archival of historical logs.
- Event streaming.

These enhancements are intentionally excluded to keep the implementation focused on the assignment requirements.

---

# 14. Summary

The database is intentionally simple.

Instead of introducing unnecessary entities or relationships, the design focuses on one responsibility:

> Maintaining a trustworthy, append-only audit trail whose integrity can be verified at any time.

This simplicity keeps the implementation maintainable while satisfying the functional and security requirements of the assignment.
